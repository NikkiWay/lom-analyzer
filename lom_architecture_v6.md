# Архитектура и методология информационно-аналитической системы раннего предупреждения об информационных рисках на основе анализа лидеров общественного мнения в ВКонтакте

### Версия 6.0 (финальная имплементационно-ориентированная спецификация)

---

## 1. Введение

Настоящий документ является финальной архитектурно-методологической спецификацией десктопного приложения, разрабатываемого в рамках дипломной работы по направлению «Информационная аналитика». Документ самодостаточен: он не требует обращения к предыдущим версиям и содержит все формулы, процедуры, политики обработки пограничных случаев, сценарии сбоев и технические решения, необходимые для перехода в фазу декомпозиции задач и непосредственной имплементации MVP.

Назначение системы — автоматизированная идентификация лидеров общественного мнения (ЛОМ) в заданной тематической нише ВКонтакте, построение агрегированной аналитической модели каждого значимого актора и раннее предупреждение аналитика об изменении структуры дискурса, появлении новых акторов и аномальных сдвигах в содержании публичной коммуникации.

Методологическое ядро образуют шесть конструкций, работающих в связке:

1. **Двухмерная модель влияния** — раздельная оценка структурного (базового) и ситуационного (событийного) положения актора.
2. **Двухпроходная временная схема** — раздельные окна для оценки «фонового» состояния актора (baseline) и для детекции событий (current), устраняющая системную ошибку посмертного определения ролей.
3. **Детерминированное разрешение двойственной шкалы** — матрица комбинирования сессионных и референсных ролей 4×2 без неопределённости интерпретации.
4. **Робастные статистические методы** — Theil-Sen регрессия для γ и ортогонализации T; Huber M-estimator для агрегированной тональности актора; Robust Sigmoid для нормализации; MAD-based диагностика качества регрессий.
5. **Полная формализация неопределённости** — каждое число сопровождается либо доверительным интервалом (двухуровневый / блочный bootstrap), либо явным флагом качества; ни один автоматический вывод не используется без явного признания его надёжности.
6. **Атомарные режимы обработки** — отсутствие смешанных состояний NLP; интерактивное восстановление при сбоях с явным выбором пользователя.

Документ ориентирован на реализацию: все ранее обнаруженные места декларативной неоднозначности (политика γ-рассогласования, вычисление референсного порога, confidence для пограничных ролей, bootstrap-процедуры, взаимодействие защитных механизмов, сценарии сбоев UX) получили конкретные процедуры и формулы.

## 2. Назначение системы

### 2.1. Предметная область и прикладная задача

ЛОМ выступают ранними интерпретаторами событий: их фреймирование способно становиться доминирующим в публичной дискуссии. Анализ публикаций ЛОМ, масштабов их распространения и аномальных изменений поведения позволяет обнаруживать сигналы информационных рисков раньше, чем сплошной мониторинг общего потока публикаций.

Прикладной пользователь — аналитик или менеджер (PR, GR, региональный коммуникационный блок), которому необходимо:

- структурировать информационное поле по выбранной теме и региону;
- идентифицировать ключевых акторов, задающих рамку обсуждения;
- различать устойчивое публичное влияние актора и его ситуационную вовлечённость в конкретную тему;
- получать интерпретируемые ранние сигналы об информационных рисках с количественной оценкой надёжности;
- поддерживать агрегированный профиль актора (модель персоны) с историей и доверительными интервалами;
- обеспечивать воспроизводимость и аудируемость результатов.

### 2.2. Отличия от типовых решений

Система отличается от типовых мониторингов настроений и инструментов с единым интегральным индексом влияния следующими свойствами:

1. **Разделение устойчивого и ситуационного.** Двухмерная модель исключает смешивание двух качественно разных типов влияния.
2. **Раздельные временные окна.** Ретроспективная оценка ролей основана только на baseline; детекция событий — только на current; системная ошибка look-ahead bias устранена.
3. **Формализованная неопределённость.** Каждый вывод сопровождается CI или флагом; пороги уверенности явно определяют, какие автоматические рекомендации блокируются.
4. **Прозрачная воспроизводимость.** Все параметры сессии, статистики нормализации, версии моделей, пользовательские решения, флаги γ-рассогласования, ортогонализации и стабильности — в AuditLog и `AnalysisSession`.
5. **Атомарность NLP-режимов.** Отсутствие частично-обработанных состояний; интерактивный выбор при сбоях вместо автоматической инвалидации.
6. **Двойная шкала с детерминированным разрешением.** Сессионные и референсные роли комбинируются через матрицу 4×2, не оставляющую downstream-логике никакой неопределённости.
7. **Робастная статистическая основа.** Theil-Sen и Huber вместо OLS и простого среднего устойчивы к типичным для соцсетей выбросам (боты, вирусные всплески).

### 2.3. Границы применимости

Система является ретроспективным аналитическим инструментом. Она **не предназначена**:

- для автоматического исполнения управленческих действий (все рекомендации — проекты для ревью);
- для real-time мониторинга (временное разрешение — дневное);
- для профилирования физических лиц вне контекста публичного дискурса;
- для атрибуции скрытой политической позиции (stance) — из-за ограниченной достижимой точности (~30%).

## 3. Цели, задачи и границы применения

### 3.1. Основная цель

Повышение качества подготовки аналитических и управленческих решений в области связей с общественностью за счёт объективизации картины информационного поля, формализации понятия ЛОМ и количественной оценки информационных рисков.

### 3.2. Задачи

1. Автоматизированный сбор публичных данных ВКонтакте в рамках заданной тематики и региона.
2. Фильтрация текстов по тематической релевантности с обязательной валидацией аналитиком по двум выборкам (стратифицированная 30 + случайная 40).
3. Дедупликация публикаций.
4. Раздельный расчёт $I_{base}$ и $I_{event}$ с двухпроходной временной схемой.
5. Ролевая классификация через детерминированную матрицу 4×2.
6. Построение модели персоны.
7. Контент-анализ (тональность с Huber-агрегацией, ключевые термы, визуальная активность).
8. Детекция аномалий с учётом роли, сезонности, праздников и защиты от рутинных публикаций.
9. Формирование рискового балла с декомпозицией.
10. Обеспечение воспроизводимости, аудируемости, правовой корректности.

## 4. Основные принципы проектирования

1. **Разделение устойчивого и ситуационного.**
2. **Разделение временных окон** (baseline vs current).
3. **Однонаправленный поток данных** с чекпоинтами между этапами.
4. **Формализованная неопределённость** (CI + флаги).
5. **Атомарные NLP-режимы** (FULL или FALLBACK_ONLY).
6. **Интерактивное восстановление** при критических сбоях (без автоматической инвалидации при > 50% прогресса).
7. **Единственность активной аналитической сессии** на экземпляр приложения.
8. **Явная идентификация источников неопределённости**, не охваченных CI (через флаги).
9. **Инверсия зависимостей** (Koin).
10. **Checkpointing на каждом шаге**, включая шаги 18 (пересчёт $M_{reach}$) и 11 (пересчёт нормализации после discovery).
11. **Прозрачность всех решений** — AuditLog из ~35 типов событий.
12. **Правовая и этическая встроенность** — шифрование, санитизация логов, retention.
13. **Устойчивость к выбросам** — Theil-Sen + Huber + MAD-based диагностика.
14. **Учёт временного контекста** — сезонность + праздники РФ.
15. **Детерминированное разрешение противоречий** — без downstream-неопределённостей.

## 5. Общая концепция системы

### 5.1. Аналитическая логика

- **Объекты измерения:** актор, публикация, реакции.
- **Уровни анализа:** пост → актор → день → сессия.
- **Временные рамки:** baseline (для ролей) + current (для аномалий).
- **Основные объекты:** $I_{base}$, $I_{event}$, роль, тональность на трёх уровнях, аномалия, рисковый сигнал, модель персоны.

Каждый аналитический объект имеет:

- определение (формулу/процедуру);
- уровень вывода;
- основание (свидетельства);
- правило разрешения противоречий;
- оценку уверенности (CI / confidence / флаг);
- границы применимости.

### 5.2. Что система детектирует и выводит

**На уровне поста:**
- тематическая релевантность (бинарно, с комбинированным score L1 + L2 и порогом θ);
- тональность ({POSITIVE, NEUTRAL, NEGATIVE} + score + bootstrapAgreement для критических);
- оригинальность (ORIGINAL / REPOST_WITH_COMMENT / PURE_REPOST / DETECTED_COPY / MEDIA_ONLY);
- принадлежность к dedup-группе;
- праздничный статус (isHoliday).

**На уровне актора (baseline-окно):**
- $I_{base}^{hist}$ + CI (двухуровневый bootstrap);
- $I_{event}^{hist}$ + CI (одноуровневый bootstrap по постам);
- сессионная роль, референсная роль, комбинированная роль;
- confidence комбинированной роли;
- флаги качества.

**На уровне актора (current-окно):**
- $I_{event}^{curr}$ + CI;
- агрегированная Huber-тональность + CI;
- VAR (Visual Activity Ratio);
- участие в аномалиях.

**На уровне дня:**
- дневной объём тематических публикаций (с сезонно-нормированным вариантом);
- средняя тональность дня (по ≥ 3 постам);
- z-score объёма / тональности с rolling окном 7 дней и учётом сезонности / праздников;
- аномалии VOLUME_SPIKE / TONE_SHIFT_NEGATIVE / TONE_SHIFT_POSITIVE / GIANT_ACTIVATION.

**На уровне сессии:**
- рисковый балл $R$ + CI (блочный bootstrap);
- доминирующий триггер и проект рекомендации;
- Session Quality Score;
- hard gates.

### 5.3. Что система не утверждает

- скорость набора реакций (однократный сбор);
- каскадное распространение 2+ уровней (ограничение API);
- истинные политические позиции авторов;
- причинно-следственные связи между событиями и публикационной активностью.

### 5.4. Что система утверждает с заявленной надёжностью

- ретроспективную картину дневной активности;
- относительную позицию актора в $(I_{base}, I_{event})$ в рамках сессии с CI;
- абсолютную позицию актора по оси $I_{base}$ относительно референсной базы при подтверждённой репрезентативности референса и согласованных параметрах γ;
- наличие дневных аномалий с severity и CI;
- рисковый балл сессии с декомпозицией и CI.

## 6. Общая архитектура

### 6.1. Выбранное решение

**Kotlin/Compose Desktop** — UI, бизнес-логика, хранилище, оркестрация.

**Встроенный Python-сидекар** — FastAPI на `127.0.0.1`, случайный свободный порт из 8300–8399, shared secret в заголовке `X-Auth-Token`. Конкурентность: семафор 4 одновременных запроса от Kotlin, uvicorn `--workers 1 --loop uvloop`.

Обе реализации NLP взаимозаменяемы через интерфейс `NlpService`.

### 6.2. Обоснование

1. Kotlin — основной язык и UI-стек.
2. Python — зрелая русскоязычная NLP-экосистема.
3. HTTP с shared secret — упрощает отладку и исключает использование сервиса сторонними процессами.
4. Fallback на Kotlin-only — гарантия работоспособности.

Пути эволюции (вне MVP): ONNX Runtime for JVM + Kotlin-лемматизатор; миграция на PostgreSQL; современные энкодеры (см. §29).

### 6.4. Разделение на процессы и потоки

- **UI-поток** — отрисовка Compose, обработка событий.
- **IO-диспетчер (Kotlin)** — до 3 одновременных VK-запросов.
- **Default-диспетчер (Kotlin)** — CPU-bound (dedup, нормализация, bootstrap с параллелизацией).
- **Python-процесс** — NLP-инференс (семафор 4 запроса).
- **Retention-поток** — при запуске проверяет просроченные сессии.

**Единственность аналитической сессии.** `ActiveSessionRegistry` — thread-safe синглтон с состояниями {IDLE, ANALYZING, RECOVERY_AWAITING}. Состояние `RECOVERY_AWAITING` означает, что аналитическая сессия активна, но приостановлена в ожидании пользовательского выбора в `PythonRecoveryDialog`. Попытка старта второй аналитической сессии при состоянии ANALYZING или RECOVERY_AWAITING возвращает доменную ошибку `ANOTHER_SESSION_ACTIVE` с именем текущей сессии, предложениями (дождаться / отменить / поставить в очередь).

### 6.5. Recovery UX при сбоях Python-сидекара

Политика сбоев:

1. Первая сетевая ошибка / HTTP 5xx / таймаут — повторная попытка с backoff (2s, 4s, 8s).
2. Вторая-третья — auto-restart процесса Python (до 3 попыток суммарно, с ожиданием 5s между попытками).
3. После трёх неуспешных auto-restart — `PythonServiceManager.onPermanentFailure()`; `ActiveSessionRegistry` переводит сессию в `RECOVERY_AWAITING`; UI показывает **модальный диалог** с тремя опциями:
    - **«Подождать и повторить запуск»** — пользователь может вручную ещё раз попытаться; возвращает сессию в ANALYZING и пытается подключиться к Python.
    - **«Перейти в режим FALLBACK_ONLY»** — все ранее Python-обработанные посты (помеченные `topicRelevanceMethod == L1_PLUS_L2` или `sentiment.method` содержащим `DOSTOEVSKY`) инвалидируются; сессия переводится в `nlpMode = FALLBACK_ONLY`; пайплайн перезапускается с текущего этапа с пересчётом через локальные методы.
    - **«Отменить сессию»** — статус сессии → `CANCELLED`; сырые собранные данные сохраняются в БД и могут быть использованы в новой сессии.

**Сценарий закрытия приложения во время активного `RECOVERY_AWAITING`.** При закрытии приложения в этом состоянии сессия автоматически переводится в `PAUSED_PENDING_RECOVERY` (отдельный статус) и сохраняется в БД. При следующем запуске приложения такая сессия видна в списке с отметкой «ожидает решения о восстановлении»; открытие сессии автоматически показывает `PythonRecoveryDialog`, и пользователь делает выбор. До выбора сессия считается активной для `ActiveSessionRegistry` — вторая аналитическая сессия в это время не разрешается.

**Запись решения.** Выбор пользователя записывается в таблицу `RecoveryChoice` (с полями sessionId, pythonFailureCount, pipelineStage, choice, timestamp, payloadJson) и в AuditLog как `PYTHON_RECOVERY_CHOICE`. При прогрессе пайплайна > 50% при выборе FALLBACK пользователь дополнительно информируется о объёме пересчитываемой работы.

### 6.6. Атомарность NLP-режимов

В течение одной **фазы работы сессии** режим фиксирован: FULL или FALLBACK_ONLY. Записанный в `AnalysisSession.nlpMode`, он не меняется без явного переключения через `PythonRecoveryDialog`.

**Смена режима между запусками приложения.** Если сессия была закрыта в режиме FULL, а при повторном открытии Python-сидекар недоступен, `SessionManager` сравнивает `AnalysisSession.nlpMode == FULL` с реальной доступностью Python. При несоответствии:

- Если сессия в статусе COMPLETED — результаты считаются замороженными; открыть в режиме просмотра можно (без пересчёта).
- Если сессия в ANALYZING / PAUSED_PENDING_RECOVERY и требует продолжения пайплайна — показывается `PythonRecoveryDialog`: продолжить в FALLBACK_ONLY (с инвалидацией Python-обработанных данных) или отменить.

Это гарантирует, что атомарность не нарушается между запусками.

## 8. Сквозной сценарий работы системы

35 шагов с явными чекпоинтами (обозначены ⧉).

1. **Запуск приложения.** `SingleInstanceLock` → `MasterPasswordDialog` → `TokenVault` инициализируется → Koin + retention поток → RetentionManager soft-delete просроченных.
2. **Создание/открытие сессии.** `SessionManager` → `ActiveSessionRegistry.canStart()` проверка при переходе в ANALYZING.
3. **Авторизация VK.** `TokenVault.get()` или OAuth в WebView → немедленная шифровка.
4. **Инициализация NLP.** `PythonServiceManager.start()` с случайным портом и shared secret → `NlpServiceSelector` определяет FULL / FALLBACK_ONLY.
5. **Валидация ресурсов.** `ResourceLoader` читает reference_base (SHA-256 + age), holidays, sentilex, test_corpus.
6. **Перевод в ANALYZING** через `ActiveSessionRegistry.startAnalysis(sessionId)`.
7. **Сбор baseline** (`BaselineCollector`). ⧉ checkpoint per-resource.
8. **Сбор current** (`CurrentCollector`). ⧉ checkpoint per-resource.
9. **Сбор репостеров** (`ReposterCollector`) для тематических постов с `0 < reposts ≤ 200`.
10. **Discovery** (`DiscoveryEngine`). До 30 кандидатов по DPS; сокращённый baseline 30 дней + current для них. ⧉
11. **Пересчёт нормализации при необходимости** (при ≥ 5 новых discovery-авторов). Сохранение старых статистик в чекпоинт `NORM_STATS_BEFORE_DISCOVERY` для возможного rollback. ⧉
12. **Предобработка** всех постов обоих окон.
13. **Тематическая фильтрация** L1 + L2. ⧉
14. **Валидация аналитиком** (30 стратифицированных + 40 случайных с Bayes Beta priors). Корректировка θ, стоп-фраз → пересчёт шага 13.
15. **Дедупликация** (exact hash + bounded Jaccard ±72ч). ⧉
16. **Классификация оригинальности.**
17. **Калибровка γ** (`GammaCalibrator` через Theil-Sen на baseline-постах после дедупликации; клип $[0.25, 0.65]$; MAD-based диагностика; bootstrap 1000 для CI; `ReferenceGammaValidator` определяет флаг рассогласования).
18. **Пересчёт $M_{reach,i}$** для постов с аппроксимацией — с финальным $\widetilde{F}$. ⧉
19. **Baseline scoring** с двухуровневым bootstrap → $I_{base}^{hist}$ + CI, $I_{event}^{hist}$ + CI, $CV_{IQR}$. ⧉
20. **Решение об ортогонализации T** (`OrthogonalizerT` через Theil-Sen; MAD-based goodness-of-fit; bootstrap 500 для p-value; условие: $R^2_{MAD} \geq 0.05$ AND bootstrap p < 0.05 AND $\hat{\beta}_1 > 0$).
21. **Current scoring.** $I_{base}^{curr}$, $I_{event}^{curr}$ + CI. $V_{raw}^{(k)}$ с $k_{window}$-нормировкой для discovery.
22. **Reference calibration.** `ReferenceCalibrator`:
    - проверка SHA-256 и возраста референса;
    - если возраст > 12 мес или SHA-256 не совпадает → $I_{base}^{abs} = NULL$, флаг `REFERENCE_UNAVAILABLE`;
    - иначе — расчёт с политикой γ-рассогласования (см. §17). ⧉
23. **Ролевая классификация.** Сессионная + референсный квадрант → `RoleCombinator` → комбинированная роль + `roleCombinationFlag` + confidence (для `BASELINE_UNKNOWN` confidence = 0 по определению). ⧉
24. **Опциональный GMM-режим** при `roleMode = GMM` и $n_{eff} \geq 50$.
25. **Контент-анализ.** Тональность (FULL или FALLBACK) всех тематических постов; для GIANT-кандидатов — `SentimentBootstrap`; TF-IDF термы; VAR. Huber-агрегация тональности на уровне актора и дня.
26. **Account flags** (`AccountFlagDetector`).
27. **Сезонная нормализация** (`SeasonalityNormalizer` с `HolidayCalendar`).
28. **Детекция аномалий.** VOLUME_SPIKE, TONE_SHIFT_{NEG, POS}, GIANT_ACTIVATION (с защитой от рутинных постов). ⧉
29. **Дедупликация аномалий** (ABSORBED_BY_GIANT).
30. **Рисковый балл.** `RiskScorer` + `BlockBootstrap` → $R$ + CI + уровень + BORDERLINE-флаги. `RecommendationEngine` → проект рекомендации. ⧉
31. **Построение персон** (`PersonaAggregator`).
32. **Session Quality Score** с проверкой hard gates. Блокировка авторекомендаций при провале gate.
33. **Persist.** `ActiveSessionRegistry.endAnalysis(sessionId)`.
34. **Публикация в UI.**

### 9.3. Discovery-проход: формализация

Три правила кандидатов:

1. **Высокий охват:** есть оригинальный тематический пост с `reposts ≥ 50` в current-окне.
2. **Массовый репостер:** репостер ≥ 5 разных оригинальных тематических постов от ≥ 3 разных seed-авторов.
3. **Контекстный вклад:** упомянут ≥ 3 раз в текстах seed-постов (NER в FULL; в FALLBACK правило применяется с пониженным весом, т.к. regex NER для русских имён имеет accuracy ~60–70%).

**Discovery Priority Score:**

$$\text{DPS}_a = \ln(1 + F_a) + \sum_{p \in P_a^{seen}} \ln(1 + M_{reach,p})$$

Топ-30. Для discovery-авторов — baseline 30 дней.

## 11. Методология извлечения тем, сущностей и аналитических объектов

### 11.1. Тематическая релевантность

Бинарное утверждение «пост принадлежит теме» — латентное свойство, аппроксимируемое двумя независимыми оценками (L1 n-грамм + L2 семантический) с контролем через валидацию аналитиком.

### 11.2. L1: N-грамм-фильтр

Excluded применяется первым; если сработала — $l_1 = 0$. Иначе:

$$l_1 = \min(\text{primary\_hits} + 0.3 \cdot \text{secondary\_hits}, 3) / 3$$

### 11.3. L2: Семантический скоринг (FULL)

rubert-tiny2 эмбеддинги; эталон — среднее 3–10 reference-текстов; cosine similarity в $[0, 1]$.

В FALLBACK_ONLY L2 отсутствует.

### 11.4. Комбинированный скор

- FULL: $\text{score} = 0.4 l_1 + 0.6 \cdot sem$, $\theta = 0.30$.
- FALLBACK_ONLY: $\text{score} = l_1$, $\theta_{l_1} = 0.33$.

### 11.5. Применение к baseline и current

Фильтр применяется к обоим окнам — явное требование для корректного $I_{event}^{hist}$.

### 11.6. Извлечение сущностей

NER через `natasha` (FULL) или regex (FALLBACK, с пониженной уверенностью). ORG, LOC, PER → `ProcessedText.entitiesJson`.

### 11.7. TF-IDF

Топ-10 термов на актора из тематических постов; IDF на сессионной коллекции.

### 11.8. Визуальная активность

$$VAR_a = \frac{\#\{p : containsMedia\}}{\#P_a}$$

**Вторичный признак.** Не влияет на $I_{base}$ или $I_{event}$; отображается в карточке персоны и в SQS. Ограничение: визуально-доминирующие акторы могут быть недооценены по текстовым метрикам (документированное ограничение).

## 12. Методология определения позиции, отношения, реакции и тональности

### 12.1. Различаемые типы

1. **Тональность поста** (POSITIVE / NEUTRAL / NEGATIVE + score + bootstrap).
2. **Реакция аудитории** (count-метрики).
3. **Фрейм** — только агрегированно, после валидации.
4. **Позиция (stance)** — не поддерживается.

### 12.2. Тональность

**FULL:** `dostoevsky` (литературная accuracy ~75–80% на отзывах; для VK-корпусов **обязательна собственная валидация**, см. §25.2).

**FALLBACK:** RuSentiLex + обработка отрицания (окно 1–3 токена: не, ни, нет, без, нельзя, никогда — инвертируют полярность).

$$\text{score} = \frac{n_{pos} - n_{neg}}{n_{pos} + n_{neg} + 1}$$

Категоризация: 0 лексем → NO_LEXICON_MATCH; $|score| < 0.15$ → LOW_CONFIDENCE; иначе POS/NEG.

### 12.3. Bootstrap для критических постов

Только для постов, потенциально активирующих `GIANT_ACTIVATION`:

- **FALLBACK:** 20 вариантов с варьированием 3 словарей негаторов × 3 окна × 3 порогов LOW_CONF.
- **FULL:** 5 запусков dostoevsky с разными random_seed + 15 вариантов постобработки.

`bootstrapAgreement` = доля согласованности. При < 0.7 → `UNSTABLE`, не используется в severity.

### 12.4. Huber M-estimator для тональности актора

Итерационная процедура для $\{s_i\}$ (тональности тематических постов актора с весами по оригинальности):

1. $\hat{\mu}_0 = \text{median}(s_i)$.
2. $w_i^{(t)} = \min(1, k / |s_i - \hat{\mu}^{(t)}|)$, $k = 1.345$.
3. $\hat{\mu}^{(t+1)} = \sum w_i^{(t)} s_i / \sum w_i^{(t)}$.
4. Сходимость при $|\hat{\mu}^{(t+1)} - \hat{\mu}^{(t)}| < 10^{-4}$; лимит итераций 20.
5. CI — bootstrap 200 по постам.

Флаг `AUTHOR_TONE_MIXED` при SD($s_i$) > 0.5 — актор демонстрирует неоднородную эмоциональную окраску (возможно бимодальное распределение; Huber-оценка в таком случае информативна только как мера центральной тенденции).

### 12.5. Тональность дня

Требование ≥ 3 тематических постов/день; Huber M-estimator; иначе `TONE_DAY_SKIPPED`.

### 12.6. Реакция аудитории

$$r_i = likes_i + 2 \cdot reposts_i + 1.5 \cdot comments_i$$

### 12.7. Фрейм-классификация

Только агрегированно, на отдельной вкладке UI, после собственной валидации (40–50 постов) с accuracy ≥ 50%. Не влияет на $R$ и роли.

### 12.8. Stance — явный отказ.

### 13.1. Уровни связи

- Пост → тема (бинарно через §11.4).
- Пост → тональность (§12.2).

## 14. Компоненты $I_{base}$ и $I_{event}$

### 14.1. $I_{base}^{hist}$

$$I_{base}^{hist} = 0.55 \cdot A_{norm} + 0.45 \cdot E_{norm}$$

#### 14.1.1. Компонент A: Аудитория

$$A_{raw} = \ln(1 + F)$$

Для закрытых профилей: $F_{imputed} = Q_{25}(F_{open})$; флаг `ESTIMATED_CONSERVATIVE`; импутированные исключаются из расчёта медианы/IQR. Ручной override возможен (AuditLog).

#### 14.1.2. Компонент E: Engagement Density

$$\bar{r}_a = \frac{1}{\max(N_{all,a}, 1)} \sum_i r_i$$

$$E_{raw,a} = \ln(1 + \bar{r}_a) - \gamma \cdot \ln(1 + F_a)$$

Агрегация $\bar{r}$ ведётся по всем постам baseline-окна **после дедупликации** (на кросс-постах одного автора $N_{all}$ учитывает схлопнутую группу как 1 пост, с $r_i$ равным максимуму по группе для реакций).

**Калибровка γ через Theil-Sen + MAD-based диагностика:**

Шаги процедуры:

1. Выборка активных авторов $A^* = \{a : F_a > 100, N_{all,a} > 5\}$ из baseline (после дедупликации).
2. Точки $\{(x_a, y_a)\} = \{(\ln F_a, \ln \bar{r}_a) : a \in A^*\}$.
3. **Theil-Sen:** $\hat{\beta}_1 = \text{median}\{(y_j - y_i) / (x_j - x_i) : i \neq j\}$; $\hat{\beta}_0 = \text{median}(y_i - \hat{\beta}_1 x_i)$.
4. $\hat{\gamma} = -\hat{\beta}_1$.
5. Клип: $\hat{\gamma} \in [0.25, 0.65]$; выход → граничное значение + флаг `GAMMA_CLIPPED`.
6. **MAD-based диагностика качества:**

   $$\text{MAD}_{residual} = \text{median}(|y_i - \hat{\beta}_0 - \hat{\beta}_1 x_i|)$$

   $$\text{MAD}_{total} = \text{median}(|y_i - \text{median}(y)|)$$

   $$R^2_{MAD} = 1 - (\text{MAD}_{residual} / \text{MAD}_{total})^2$$

7. **Bootstrap CI:** 1000 итераций. В каждой — выборка с возвращением пар $(x_a, y_a)$ из $A^*$, пересчёт Theil-Sen; 5/95 перцентили $\hat{\gamma}$ → CI.
8. При $|A^*| < 20$ → fallback на OLS; при $R^2_{MAD} < 0.05$ AND число активных < 20 → $\gamma = 0.45$ + флаг `GAMMA_FALLBACK`.
9. Результат (оценка, $R^2_{MAD}$, 95% CI, флаги) → `AnalysisSession.normStatsJson.gamma`.

### 14.2. $I_{event}$

$$I_{event} = w_T T^\star_{norm} + w_V V_{norm}^{(k)} + w_S S_{norm} + w_O O_{norm}$$

**Весовые наборы:**

- Набор A (ортогонализация применена): $(0.20, 0.30, 0.30, 0.20)$.
- Набор B (ортогонализация не применена): $(0.15, 0.30, 0.35, 0.20)$.

Выбор фиксируется в `AnalysisSession.eventWeightsVariant`.

#### 14.2.1. Компонент T с leave-one-out prior

$$p_a = \frac{\sum_{a' \neq a} N_{topic,a'}}{\sum_{a' \neq a} N_{all,a'}}$$

$$T_{raw,a} = \frac{N_{topic,a} + \alpha p_a}{N_{all,a} + \alpha}, \quad \alpha = 5$$

Защита: $T_{raw} = \min(T_{raw}, 1.0)$.

**Ортогонализация на V через Theil-Sen + MAD-based диагностика:**

1. Точки $\{(\ln(1 + N_{topic,eff,a}), T_{raw,a})\}$ по всем авторам.
2. Theil-Sen → $\hat{\beta}_0, \hat{\beta}_1$.
3. MAD-based $R^2_{MAD}$ (аналогично §14.1.2).
4. **Bootstrap p-value для значимости $\hat{\beta}_1$:** 500 итераций permutation-теста. На каждой — переставляется y-координата случайно; пересчитывается Theil-Sen на перемешанных данных; p-value = доля перестановочных $|\hat{\beta}_1^{perm}|$, превышающих реальное $|\hat{\beta}_1|$.
5. **Условие применения:** $R^2_{MAD} \geq 0.05$ AND p-value < 0.05 AND $\hat{\beta}_1 > 0$.
6. Если выполнено: $T^\perp_{raw,a} = T_{raw,a} - \hat{\beta}_0 - \hat{\beta}_1 \ln(1 + N_{topic,eff,a})$; набор A; флаг `T_ORTHOGONALIZED = true`.
7. Иначе: $T^\star = T$; набор B; флаг `T_ORTHOGONALIZED = false`.

#### 14.2.2. Компонент V с $k_{window}$-нормировкой

$$k_{window,a} = \frac{\text{baseline\_days\_actual}_a}{60}$$

$$V_{raw,a}^{(k)} = \ln\left(1 + \frac{N_{topic,eff,a}}{k_{window,a}}\right)$$

**Допущение стационарности.** Коррекция $k_{window}$ корректна при предположении, что темп тематических публикаций актора приблизительно стационарен в baseline-окне. Это явное методологическое допущение. Для акторов с сильной нестационарностью (длинные периоды неактивности + всплески) оценка $I_{event}^{hist}$ может быть смещена. Флаг `POSSIBLY_NONSTATIONARY` устанавливается, если коэффициент вариации $N_{topic}/\text{неделя}$ в baseline > 1.5 (сигнал высокой нестационарности).

#### 14.2.3. Компонент S: Dissemination Reach

$M_{reach,i} = \sum_{r \in R_i} F_r$; полный сбор при `reposts ≤ 200`; иначе аппроксимация $\approx reposts_i \cdot \widetilde{F}$.

$\widetilde{F}$ — медиана `followers_count` по всем собранным репостерам сессии при ≥ 30 наблюдениях; иначе $\widetilde{F}^{ref}$ из референса.

**Пересчёт $M_{reach,i}$** для аппроксимированных постов — шаг 18 §8 с чекпоинтом.

$$S_{raw,a} = \frac{1}{\max(N_{topic,eff,a}, 1)} \sum_{i \in P_{topic,a}} \ln(1 + M_{reach,i})$$

Также инвариантно к длительности окна при допущении стационарности (см. §14.2.2).

#### 14.2.4. Компонент O: Оригинальность

| Тип | Условие | Вес |
|-----|---------|-----|
| ORIGINAL | Нет copy_history, ownTextLength ≥ 20, не DETECTED_COPY | 1.0 |
| REPOST_WITH_COMMENT | Есть copy_history, ownTextLength ≥ 30 | 0.5 |
| PURE_REPOST | Есть copy_history, ownTextLength < 30 | 0.0 |
| DETECTED_COPY | SHA-256 совпадает с более ранней публикацией другого автора | 0.0 |
| MEDIA_ONLY | Нет copy_history, ownTextLength < 20, containsMedia | 0.25 |

$$O_{raw,a} = \frac{\sum_i w_i + 0.5}{N_{topic,eff,a} + 1}$$

## 15. Единая политика нормализации

### 15.1. Robust Sigmoid

$$z = \frac{x_{raw} - \text{median}(X)}{\text{IQR}(X) + \epsilon}, \quad x_{norm} = \sigma(z) = 1 / (1 + e^{-z})$$

### 15.2. Каскад fallback

| Условие | Действие |
|---------|----------|
| $n_{eff} \geq 30$, $IQR \geq 0.001$ | Стандартный |
| $IQR < 0.001$ | $\text{spread} = Q_{90} - Q_{10}$ |
| $\text{spread} < 0.001$ | 0.1 |
| $n_{eff} < 30$ | Bootstrap IQR 1000 итераций; сохранение $CV_{IQR}$; при $CV_{IQR} > 0.25$ → флаг `NORM_UNSTABLE` |
| $n_{eff} = 1$ | $x_{norm} = 0.5$ |

### 15.3. Последствия флагов стабильности

- $CV_{IQR} \leq 0.25$: стандартная обработка.
- $0.25 < CV_{IQR} \leq 0.35$: `NORM_UNSTABLE`; акторы с $|I - \tau| / q < CV_{IQR}$ → `ROLE_UNSTABLE`.
- $0.35 < CV_{IQR} \leq 0.5$: `NORM_UNSTABLE_SEVERE`; все с confidence < 0.5 → `UNCLASSIFIED`; рекомендации «среднее качество».
- $CV_{IQR} > 0.5$: hard gate — рекомендации **блокируются полностью**.

### 15.4. Пересчёт при discovery

При ≥ 5 новых discovery-авторов:
1. Сохранение предыдущих медиан/IQR в checkpoint `NORM_STATS_BEFORE_DISCOVERY`.
2. Пересчёт медиан/IQR/$CV_{IQR}$ на расширенной выборке.
3. Применение ко всем LomScore.
4. Событие `NORM_STATS_RECOMPUTED_POST_DISCOVERY` → AuditLog.
5. Rollback через 30 дней возможен.

### 15.5. Log-преобразование

Счётные ($F$, $\bar{r}$, $N_{topic}$, $M_{reach}$) → $\ln(1+x)$ до нормализации. Доли ($T^\star$, $O$) — без. Величины с встроенным логом ($E$, $S$) — без.

### 15.6. Воспроизводимость

Все `median`, `IQR`, `spread`, `n_eff`, `CV_IQR`, флаги → `AnalysisSession.normStatsJson`.

## 16. Bootstrap и оценка уверенности

### 16.1. Двухуровневый bootstrap для $I_{base}$

- **Внешний (по авторам):** 300 итераций, выборка авторов с возвращением, пересчёт медианы/IQR.
- **Внутренний (по постам):** 100 итераций внутри каждой внешней, выборка baseline-постов автора, пересчёт $\bar{r}_a$, $E_{raw,a}$.

Итого 30 000 реализаций на автора. CI = 5/95 перцентили.

**Параметры внутри итераций:**
- γ, результаты ортогонализации — **фиксированы** на точечных оценках сессии (их вариация учитывается в sensitivity-анализе);
- leave-one-out prior $p_a$ пересчитывается внутри каждой внешней итерации;
- референсные статистики — фиксированы.

**Охваченные источники:** состав выборки, нестабильность медианы/IQR, дисперсия $\bar{r}$ актора.

**Не охваченные (фиксируются отдельными флагами):**
- Актуальность $F$ (`AUDIENCE_TIMESTAMP`);
- Закрытые профили (`ESTIMATED_CONSERVATIVE`);
- Дрейф референса (`REFERENCE_FRESHNESS`);
- γ-рассогласование (`REFERENCE_GAMMA_DIVERGENCE`);
- Нестационарность (`POSSIBLY_NONSTATIONARY`).

### 16.2. Bootstrap для $I_{event}$

Одноуровневый по постам: 500 итераций. $N_{topic,eff}$, $S_{raw}$, $O_{raw}$, $T_{raw}$ пересчитываются. Leave-one-out prior — внутри. γ, ортогонализация — фиксированы.

### 16.3. Блочный bootstrap для $R$

Блоки 7 дней, выборка из baseline + current, 300 итераций. Пересчёт severity и $R$. CI 5/95. BORDERLINE-маркировка.

### 16.4. $CV_{IQR}$

Интегрирован в внешний bootstrap $I_{base}$: для каждой из 300 итераций сохраняется IQR; по 300 значениям вычисляется $CV = \sigma(IQR) / \text{mean}(IQR)$.

### 16.5. Производительность

Проектная оценка при $n = 200$ авторов, 4 ядра:
- $I_{base}$ двухуровневый: ≈ 8 мин;
- $I_{event}$: ≈ 3 мин;
- $R$ block: ≈ 1 мин.

Всего ≈ 12 мин. Подтверждается benchmark-harness-ом (§25.6) до релиза v1.0. При превышении — снижение параметров bootstrap по калибровочной таблице.

При $n_{posts} < 10$ для автора или $n_{eff} < 20$ для сессии — bootstrap отключается, CI = `NaN`, UI — «CI недоступен».

### 16.6. Confidence роли

$$d_{base} = |I_{base}^{hist} - \tau^{sess}_{base}| / q^{sess}_{base}$$

$$d_{event} = |I_{event}^{hist} - \tau^{sess}_{event}| / q^{sess}_{event}$$

$$\text{confidence}_{raw} = \min(d_{base}, d_{event})$$

$$\text{confidence} = \text{clamp}\left(\text{confidence}_{raw} \cdot \frac{\sqrt{n_{eff}}}{\sqrt{n_{eff}} + 10}, 0, 1\right)$$

**Специальный случай `BASELINE_UNKNOWN`.** У такого актора $I_{event}^{hist}$ не вычислен → $d_{event}$ не определено. По определению для ролей `BASELINE_UNKNOWN` устанавливается $\text{confidence} = 0$. Это отражает максимальную неопределённость классификации при отсутствии базы. Эвристический режим `GIANT_ACTIVATION` (§19.4) применяется независимо от confidence, с фиксированным модификатором 0.5.

### 17.3. Протокол сбора референса

Выполняется разработчиком **вне пайплайна сессии**. Процедура:

1. **Стратифицированная выборка** 15000 публичных аккаунтов ВКонтакте по стратам (аудитория, регион, тип).
2. Для каждого автора сбор за 6 месяцев $F$, $N_{all}$, суммарных реакций (для $\bar{r}$).
3. Расчёт квантилей и IQR для $\ln F$, $\ln \bar{r}$.
4. **Вычисление $E_{raw,a}^{ref}$** для каждого референсного автора с γ_ref = 0.45:
   $$E_{raw,a}^{ref} = \ln(1 + \bar{r}_a) - 0.45 \ln(1 + F_a)$$
5. Квантили $E_{raw}$ → `computed_statistics_at_gamma_ref.E_raw_at_gamma_0_45`.
6. **Вычисление $I_{base,a}^{ref}$** для каждого референсного автора через самоориентированную Robust Sigmoid:
   - нормализация $A_a$: используя $\text{median}(\ln F)$ и $\text{IQR}(\ln F)$ по референсу;
   - нормализация $E_a$: используя $\text{median}(E_{raw})$ и $\text{IQR}(E_{raw})$ по референсу;
   - $I_{base,a}^{ref} = 0.55 A_{norm,a}^{ref} + 0.45 E_{norm,a}^{ref}$.
7. Квантили $I_{base,a}^{ref}$ → `computed_statistics_at_gamma_ref.I_base_at_gamma_0_45`.
8. **Референсный порог:** $\tau^{ref}_{base}$ при γ_ref — это 75-й перцентиль $I_{base}^{ref}$: `tau_base_p75_at_gamma_ref = 0.78`.
9. $\widetilde{F}^{ref}$ — медиана `followers_count` репостеров.
10. SHA-256 файла, методология в `ref_base_methodology.md`.

### 17.4. Статус референсной базы в MVP

**Явное признание.** В MVP референс поставляется как статический пример, собранный разработчиком. Абсолютная шкала — **индикативная**, не окончательная.

Обновление референса — не чаще раза в 3–6 месяцев. SessionQualityScreen отображает цветовую индикацию freshness:
- зелёный: < 3 мес;
- жёлтый: 3–6 мес;
- оранжевый: 6–9 мес;
- красный: 9–12 мес;
- > 12 мес: абсолютная шкала **отключается** автоматически.

### 17.5. Калибровка $I_{base}^{abs}$

#### 17.5.1. Политика γ-рассогласования

Сравнение $\gamma_{sess}$ и $\gamma_{ref} = 0.45$:

| Условие | Стратегия | Флаг |
|---------|-----------|------|
| $|\gamma_{sess} - 0.45| \leq 0.1$ | Прямое использование референсных квантилей $E_{raw}^{ref}$ | `OK` |
| $0.1 < |\gamma_{sess} - 0.45| \leq 0.2$ | **Пересчёт $E_{norm}^{ref}$ под сессионную γ** (см. §17.5.2) | `MILD_RECOMPUTED` |
| $|\gamma_{sess} - 0.45| > 0.2$ | Упрощённая формула: только $A_{norm}^{ref}$ | `AUDIENCE_ONLY_REFERENCE` |

#### 17.5.2. Процедура пересчёта при флаге MILD_RECOMPUTED

Поскольку `reference_base.json` хранит **сырые квантили** $\ln F$ и $\ln \bar{r}$ (не только $E_{raw}$ при γ_ref), возможна экстраполяция $E$-квантилей под актуальную γ:

Предположение приближения (разумное при умеренных расхождениях γ и приближённо-нормальном распределении $\ln F, \ln \bar{r}$): квантиль $E$ при γ_sess оценивается как

$$q_p(E_{raw}|\gamma_{sess}) \approx q_p(\ln(1+\bar{r})) - \gamma_{sess} \cdot q_p(\ln(1+F))$$

(по соответствующим перцентилям).

**Шаги пересчёта:**

1. Из `reference_base.json` извлекаются квантили $q_{25}, q_{50}, q_{75}$ для $\ln F$ и $\ln \bar{r}$.
2. Квантили $E$ при γ_sess:
   - $q_{50}(E|\gamma_{sess}) = q_{50}(\ln \bar{r}) - \gamma_{sess} \cdot q_{50}(\ln F)$;
   - $q_{25}(E|\gamma_{sess}) = q_{25}(\ln \bar{r}) - \gamma_{sess} \cdot q_{25}(\ln F)$ (примерное соответствие квантилей);
   - $q_{75}(E|\gamma_{sess}) = q_{75}(\ln \bar{r}) - \gamma_{sess} \cdot q_{75}(\ln F)$.
3. $IQR(E|\gamma_{sess}) = q_{75}(E|\gamma_{sess}) - q_{25}(E|\gamma_{sess})$.
4. Нормализация $E_{norm,a}^{ref}$ через Robust Sigmoid с этими пересчитанными медианой и IQR.
5. Также пересчитывается $\tau^{ref}_{base}$ — при новой γ_sess порог референсных p75 нужно пересчитать. Приближённо: $\tau^{ref}_{base, new} \approx \tau^{ref}_{base}(0.78)$ (так как порог определялся на ранговой структуре; структура ранговой зависимости от γ относительно слабая при умеренных расхождениях). Устанавливается флаг `REF_THRESHOLD_APPROXIMATED`.

Ограничение приближения: точно только при линейной зависимости квантилей (нормальное распределение). Для русскоязычных соцсетей распределения имеют тяжёлые хвосты, и приближение может давать погрешность 5–15% в оценке $\tau$. Это документированное ограничение; при флаге MILD_RECOMPUTED пользователь информируется.

#### 17.5.3. Формула при OK и MILD_RECOMPUTED

$$z_{A,a}^{ref} = \frac{\ln(1 + F_a) - q_{50}(\ln F)^{ref}}{IQR(\ln F)^{ref} + \epsilon}$$

$$z_{E,a}^{ref} = \frac{E_{raw,a}^{sess} - q_{50}(E|\gamma)^{ref}}{IQR(E|\gamma)^{ref} + \epsilon}$$

где $E_{raw,a}^{sess}$ вычислено с сессионной γ; квантили $E$ — прямые (OK) или пересчитанные (MILD_RECOMPUTED).

$$A_{norm}^{ref} = \sigma(z_{A,a}^{ref}), \quad E_{norm}^{ref} = \sigma(z_{E,a}^{ref})$$

$$I_{base,a}^{abs} = 0.55 A_{norm}^{ref} + 0.45 E_{norm}^{ref}$$

#### 17.5.4. Формула при AUDIENCE_ONLY_REFERENCE

$$I_{base,a}^{abs} = A_{norm,a}^{ref}$$

Компонент E исключён.

### 17.6. Референсные пороги

$$\tau^{ref}_{base} = \text{tau\_base\_p75}$$

Процедура расчёта (§17.3, шаги 6–8): для каждого референсного автора вычисляется $I_{base,a}^{ref}$ через самоориентированную Robust Sigmoid; $\tau^{ref}_{base}$ = 75-й перцентиль.

Для $I_{event}$ референсного порога нет.

$F_{p75}^{ref}$ — эвристический порог для режима `BASELINE_UNKNOWN` (§18.5); хранится в `reference_base.json.I_base_thresholds.F_p75`.

### 17.7. Версионирование

- `AnalysisSession.referenceBaseVersion` + `referenceBaseSha256`.
- Смена референса во время активной сессии блокируется: `ReferenceCalibrator` читает файл единожды и кэширует.
- Расхождение SHA-256 → сессия не открывается, требует пересчёта.

## 18. Ролевая классификация

### 18.1. Сессионная роль

| Условие | Роль |
|---------|------|
| $I_{base}^{hist} \geq \tau^{sess}_{base}$ AND $I_{event}^{hist} \geq \tau^{sess}_{event}$ | AUTHORITATIVE_LOM |
| $I_{base}^{hist} \geq \tau^{sess}_{base}$ AND $I_{event}^{hist} < \tau^{sess}_{event}$ | SLEEPING_GIANT |
| $I_{base}^{hist} < \tau^{sess}_{base}$ AND $I_{event}^{hist} \geq \tau^{sess}_{event}$ | TOPIC_DRIVER |
| $I_{base}^{hist} < \tau^{sess}_{base}$ AND $I_{event}^{hist} < \tau^{sess}_{event}$ | BACKGROUND |

$\tau^{sess}_{base} = \text{median}(I_{base}^{hist})$, $\tau^{sess}_{event} = \text{median}(I_{event}^{hist})$.

### 18.2. Референсный квадрант (по $I_{base}$)

| Условие | Квадрант |
|---------|----------|
| $I_{base}^{abs} \geq \tau^{ref}_{base}$ | HIGH_ABS_BASE |
| $I_{base}^{abs} < \tau^{ref}_{base}$ | LOW_ABS_BASE |

При `REFERENCE_UNAVAILABLE` (референс невалиден или γ-рассогласование > 0.2 со флагом AUDIENCE_ONLY_REFERENCE, где референсный порог пересчитан через аудиторию) — используется $F$-порог: `HIGH_ABS_BASE` если $F > F_{p75}^{ref}$.

### 18.3. Комбинированная роль — матрица 4×2

| Сессионная \ Референсная | HIGH_ABS_BASE | LOW_ABS_BASE |
|--------------------------|---------------|--------------|
| AUTHORITATIVE_LOM | `AUTHORITATIVE_LOM_CONFIRMED` | `AUTHORITATIVE_LOM_LOCAL` |
| SLEEPING_GIANT | `SLEEPING_GIANT_CONFIRMED` | `SLEEPING_GIANT_LOCAL` |
| TOPIC_DRIVER | `TOPIC_DRIVER_WITH_BASE` | `TOPIC_DRIVER` |
| BACKGROUND | `BACKGROUND_LARGE` | `BACKGROUND` |

### 18.4. Использование комбинированной роли детекторами

**`GIANT_ACTIVATION` срабатывает** только для: `SLEEPING_GIANT_CONFIRMED` (множитель 1.0), `SLEEPING_GIANT_LOCAL` (× 0.7), `BASELINE_UNKNOWN` с $F > F_{p75}^{ref}$ (× 0.5).

Tooltip `_LOCAL`: «Актор локально крупный только в рамках выборки. По референсу — средний. Возможно региональная специфика / нишевая тематичность / недостаточная репрезентативность. Требует контекстной проверки».

### 18.5. `BASELINE_UNKNOWN`

Автор с ≤ 3 постов в baseline:
- $I_{event}^{hist}$ не вычисляется;
- $I_{base}^{hist}$ упрощённо (только $A_{norm}$, без E); флаг `LOW_BASELINE_COVERAGE`;
- комбинированная роль = `BASELINE_UNKNOWN`;
- **confidence = 0 по определению**;
- эвристический режим GIANT_ACTIVATION при $F > F_{p75}^{ref}$ с модификатором 0.5.

### 18.6. Confidence

Согласно §16.6. Для `BASELINE_UNKNOWN` — 0. Для ролей с `ROLE_UNSTABLE` флагом — confidence × 0.5.

Пороги:
- < 0.25 — ролевые авторекомендации блокируются;
- 0.25–0.60 — средняя;
- ≥ 0.60 — высокая.

### 18.7. GMM-режим

При `roleMode = GMM` и $n_{eff} \geq 50$. 2–3 компоненты GMM на $(I_{base}^{hist}, I_{event}^{hist})$, выбор по BIC. Центры масс → роли по квадрантам. Confidence = апостериорная вероятность.

При смене режима — инвалидация и пересчёт всех зависимых артефактов; AuditLog.

## 19. Детекция аномалий

### 19.1. Типы

| Тип | Код | Описание |
|-----|-----|----------|
| VOLUME_SPIKE | VOLUME_SPIKE | Резкий рост дневного числа тематических постов |
| TONE_SHIFT_NEGATIVE | TONE_SHIFT_NEGATIVE | Резкое падение дневной Huber-тональности |
| TONE_SHIFT_POSITIVE | TONE_SHIFT_POSITIVE | Резкий рост (возможно координированный) |
| GIANT_ACTIVATION | GIANT_ACTIVATION | Тематический пост от SLEEPING_GIANT с защитой от рутинных |

### 19.2. Сезонная нормализация

#### 19.2.1. Недельный паттерн (без праздников)

$$\bar{x}_{wd} = \text{mean}\{x_d : \text{weekday}(d) = wd, d \in B \cup C, \neg \text{isHoliday}(d)\}$$

$$x_d^{norm} = \begin{cases}
x_d / \bar{x}_{wd} & \text{будний день, } \bar{x}_{wd} \geq x_{min}^{seas} \\
x_d / \bar{x}_{holiday} & \text{праздничный день, } \bar{x}_{holiday} \geq x_{min}^{seas} \\
x_d & \text{иначе (нормализация отключена для этого дня)}
\end{cases}$$

$x_{min}^{seas}$: 1.0 для volume, 0.01 для tone.

#### 19.2.2. Праздничный паттерн

$\bar{x}_{holiday} = \text{mean}\{x_d : \text{isHoliday}(d) = true\}$. Если праздничных дней в baseline ∪ current < 3, праздничная нормализация отключается; z-score для таких дней не вычисляется (аномалия не детектируется для праздничных дней). Флаг `HOLIDAY_ANOMALY_DISABLED_INSUFFICIENT_DATA`.

На временных рядах UI — `HolidayMarker`.

При baseline < 30 дней — вся сезонная нормализация отключается, флаг `SEASONALITY_DISABLED_SHORT_BASELINE`, SQS снижение $q_{temporal\_stability}$.

### 19.3. Rolling z-score

Окно $W = 7$ дней. $z_d = (x_d^{norm} - \mu_{rolling}) / (\sigma_{rolling} + \epsilon)$. Защита: $\sigma_{rolling} < \sigma_{min}^{component}$ → $z_d = 0$.

### 19.4. Severity

**VOLUME_SPIKE:** $z_d > 2.5$, $sev_{vol} = \text{clamp}(z_d / 6.0, 0, 1)$.

**TONE_SHIFT_*:** $|z_d| > 2.0$, ≥ 3 постов/день, $sev_{tone} = \text{clamp}(|z_d| / 5.0, 0, 1)$.

**GIANT_ACTIVATION с защитой от рутинных постов и уточнением взаимодействия с квадрантом:**

Условие срабатывания:
1. Комбинированная роль ∈ {SLEEPING_GIANT_CONFIRMED, SLEEPING_GIANT_LOCAL, BASELINE_UNKNOWN с $F > F_{p75}^{ref}$}.
2. Автор совершил тематический пост в current-окне.
3. **Защита от рутинных постов:** проверка $N_{topic}^{baseline}$ (число тематических постов автора в baseline-окне после дедупликации):
    - Если baseline ≥ 60 дней AND $N_{topic}^{baseline} \geq 2$ → **подавление**: аномалия не генерируется, флаг `ROUTINE_TOPIC_POSTS_IN_BASELINE` в post. Это возможно в краевых случаях, когда актор получает роль SLEEPING_GIANT квадрантной классификацией из-за того, что $I_{event}^{hist}$ слегка ниже сессионной медианы (другие авторы активнее), но уже имеет рутинную тематическую активность. Защита исключает ложные срабатывания.
    - Если $N_{topic}^{baseline} = 1$ → срабатывает с дополнительным модификатором severity × 0.8.
    - Если $N_{topic}^{baseline} = 0$ → срабатывает с полной severity (× 1.0).

**Взаимодействие с квадрантной классификацией.** Для большинства акторов с $N_{topic}^{baseline} \geq 2$ квадрант сам отнесёт их к AUTHORITATIVE_LOM или TOPIC_DRIVER (т.к. $I_{event}^{hist}$ высокое). Защита от рутинных постов — **дополнительный фильтр** для пограничных случаев, когда квадрантная классификация всё же помечает актора как SLEEPING_GIANT (например, в выборке с очень активными остальными авторами). Без этой защиты такой автор генерировал бы ложный GIANT_ACTIVATION при каждом новом тематическом посте, хотя публикует по теме регулярно. Защита — belt-and-suspenders подход, повышающий precision детектора.

Базовая severity:

$$sev_{giant}^{raw} = I_{base}^{hist}(a) \cdot (0.3 + 0.7 \cdot \text{risk\_tone})$$

$$\text{risk\_tone} = \begin{cases} \text{clamp}(-tone_{post}, 0, 1) & \text{bootstrapAgreement}_{post} \geq 0.7 \\ 0 & \text{иначе (UNSTABLE)} \end{cases}$$

**Модификаторы (перемножаются):**

| Фактор | Модификатор |
|--------|-------------|
| SLEEPING_GIANT_CONFIRMED | 1.0 |
| SLEEPING_GIANT_LOCAL | 0.7 |
| Эвристический (BASELINE_UNKNOWN + F > порог) | 0.5 |
| $N_{topic}^{baseline} = 1$ | × 0.8 (дополнительно) |
| $N_{topic}^{baseline} = 0$ | × 1.0 (дополнительно, нейтрально) |

**Confidence не входит в severity.**

### 19.5. Дедупликация аномалий

TONE_SHIFT_NEGATIVE + GIANT_ACTIVATION в один день от одного автора-гиганта: пересчёт $t_d$ без постов гиганта; если TONE_SHIFT пропадает → `suppressed = true, suppressionReason = ABSORBED_BY_GIANT`.

## 20. Рисковый балл $R$

### 20.1. Формула

$$s_{vol} = 0.30 sev_{vol}, \quad s_{tone} = 0.30 sev_{tone}, \quad s_{giant} = 0.40 sev_{giant}$$

$$R_{dom} = \max(\cdots), \quad S_{total} = s_{vol} + s_{tone} + s_{giant}$$

$$n_{active} = |\{s > 0.05\}|$$

$$R_{multi} = 1 + 1.2 \cdot (1 - R_{dom} / (S_{total} + 0.001)) \cdot \mathbb{1}[n_{active} \geq 2]$$

Максимум $R_{multi} = 1.8$ при 3 равных компонентах.

$$R = \text{clamp}(R_{dom} R_{multi}, 0, 1)$$

### 20.2. Уровни

LOW $[0, 0.15)$, MEDIUM $[0.15, 0.35)$, HIGH $[0.35, 0.55)$, CRITICAL $[0.55, 1]$. Параметризуются в `AnalysisSession.riskThresholdsJson`.

### 20.3. Confidence уровня

Block bootstrap 300 итераций даёт CI для $R$. Если CI пересекает границу → BORDERLINE.

### 20.4. Рекомендация

Триггер = argmax. Проекты в таблице §20.4 v5. Все помечены `[ЧЕРНОВИК]`.

### 22.10. Смешанная эмоциональность актора

SD > 0.5 → флаг `AUTHOR_TONE_MIXED`; Huber даёт медианную тональность, но бимодальность нужно учитывать при интерпретации.

### 22.11. Нестационарность

$CV(N_{topic}/\text{неделя}) > 1.5$ в baseline → флаг `POSSIBLY_NONSTATIONARY`; UI — предупреждение о том, что $I_{event}^{hist}$ может быть смещён.

## 24. Метрики качества сессии

### 24.1. SQS

$$Q = 0.20 q_{cov} + 0.13 q_{n} + 0.13 q_{CV} + 0.10 q_{cls} + 0.10 q_{nlp} + 0.10 q_{val} + 0.10 q_{ref} + 0.10 q_{temp} + 0.04 q_{nonstat}$$

| Компонент | Формула |
|-----------|---------|
| $q_{cov}$ | coverageRatio |
| $q_{n}$ | $\min(1, n_{eff}/100)$ |
| $q_{CV}$ | $1 - \min(1, CV_{IQR}/0.5)$ |
| $q_{cls}$ | $1 - \min(1, \text{closedRatio}/0.5)$ |
| $q_{nlp}$ | 1.0 FULL / 0.7 FALLBACK |
| $q_{val}$ | 1 если валидация / 0 иначе (бинарно) |
| $q_{ref}$ | 1.0 (<3мес) / 0.7 (3–6) / 0.4 (6–9) / 0.1 (9–12) / 0.0 (>12) |
| $q_{temp}$ | 1.0 сезонность работает / 0.8 частично (HOLIDAYS_PARTIAL_COVERAGE или < 3 праздников) / 0.5 отключена |
| $q_{nonstat}$ | $1 - \text{доля авторов с } POSSIBLY\_NONSTATIONARY$ |

### 24.2. Последствия

- $Q \geq 0.75$: высокое.
- $0.5 \leq Q < 0.75$: среднее с маркером.
- $Q < 0.5$: **блокировка рекомендаций**.

### 24.3. Hard gates

| Gate | Условие | Действие |
|------|---------|----------|
| Coverage | < 0.5 | Описательный режим, ролевой классификации нет |
| Sample size | $n_{eff} < 20$ | Статистика отключена |
| Reference | Возраст > 12 мес | Абсолютная шкала отключена |
| Norm stability | $CV_{IQR} > 0.5$ | Рекомендации блокированы |
| Validation | Не выполнена | Рекомендации блокированы |

## 25. Верификация, контроль качества и оценка корректности

### 25.1. Валидация тематического фильтра

Стратифицированная 30 + случайная 40. Bayes Beta(1,1) prior; posterior Beta(1+TP, 1+FP) для precision, Beta(1+TP, 1+FN) для recall. CI 95%.

50 постов из VK-корпуса; accuracy, macro-F1, bootstrap 500 CI. Результат в AuditLog; определяет, реалистична ли декларируемая accuracy.

### 25.3. Валидация ролей

30–50 акторов + 1–2 независимых эксперта; κ Коэна с inter-rater agreement. Мощность ±0.2 при n=30 — явно сообщается.

### 25.4. Валидация фреймов

40–50 постов, accuracy ≥ 50% — условие отображения.

### 25.5. Sensitivity harness

1. `test_corpus.json` (§9.6).

Обязательный gate к v1.0.

## 26. Правовой и этический слой

### 26.1. Нормативная база

- 152-ФЗ (ст. 6, 10).
- Пользовательское соглашение ВК и правила API.

### 26.2. Меры

1. **Локальное хранение.**
2. **Шифрование токена:** AES-GCM, ключ из мастер-пароля (PBKDF2 ≥ 100000 итераций) или OS keychain. На Linux без libsecret — обязательно мастер-пароль.
3. **Жизненный цикл токена:**
    - мастер-пароль один раз при запуске;
    - токен в памяти зашифрован;
    - новый токен через встроенный WebView OAuth;
    - переиспользуется между сессиями в рамках запуска;
    - при закрытии — `Arrays.fill(token, 0)`;
    - зашифрованный токен на диске в `<app_data>/token_vault.bin`.
4. **Хэширование PII в экспорте:** SHA-256 с сессионной солью по умолчанию; сырой экспорт — через подтверждение.
5. **Санитизация PII в логах:** `PiiSafeFormatter` (см. §27).
6. **Retention:** soft-delete через 12 мес, hard-delete через 30 дней grace.
7. **Право на забвение:** каскадное удаление автора из всех сессий.
8. **AuditLog** всех действий.
9. **Кросс-сессионное сопоставление:** невозможно из-за сессионной соли; компромисс приватности.

### 26.3. Этические ограничения

- Термины ролей — с расшифровкой в отчётах;
- Рекомендации — только `[ЧЕРНОВИК]`;
- Запрет профилирования вне публичного дискурса.

## 27. Наблюдаемость, логирование, тестирование

### 27.1. Структурированное логирование

### 27.1.1. Санитизация PII

`PiiSafeFormatter` подключён как encoder. Правила маскирования:

- `vk_id: 123456789` → `vk_id: 1****89`;
- `screen_name: ivanov123` → `screen_name: iv*****23`;
- `firstName`, `lastName` → `***`;
- Текстовые поля > 80 символов → `<redacted:{length}>`.

**Build-time проверка.** В production-сборке (gradle property `buildType=production`) флаг `DEBUG_INCLUDES_PII = true` отклоняется проверкой при старте; приложение пишет критическое сообщение и прекращает запуск.

**Unit-тесты.** Обязательная серия тестов для масок: минимум 20 тестов, покрывающих:
- поля `vk_id`, `screen_name` на верхнем уровне и во вложенных структурах;
- поля в URL-параметрах (через HTTP запросы);
- поля в строковых описаниях (через regex);
- длинные текстовые поля;

### 32.1. Методологические ограничения

1. Сессионная относительность $I_{event}$.
2. Тональность словарно ~65–68%, dostoevsky ~75–80% (литературно); обязательна собственная валидация.
3. Фрейм — только агрегированно, после валидации.
4. Stance — не поддерживается.
5. Скорость распространения невычислима.
6. $S$ — потенциальный охват.
7. При reposts > 200 — аппроксимация с пересчётом.
8. Dedup ±72ч.
9. Эвристики ботов — предупреждения.
10. Закрытые профили $Q_{25}$-импутация.
11. Каскады 2+ недоступны.
12. 10–30 сообществ / 50–200 авторов + 30 discovery (отражено в SQS).
13. γ в $[0.25, 0.65]$; политика рассогласования.
14. Baseline 30–180 дней; discovery 30 дней; $k_{window}$-нормировка.
15. Мультимодальный контент — только VAR.
16. Русский язык.
17. Референс — индикативный.
18. **Допущение стационарности** для инвариантности $T, S, O$ к длительности окна; флаг `POSSIBLY_NONSTATIONARY` при нарушении.
19. **MILD_RECOMPUTED γ-пересчёт** — приближение; погрешность $\tau^{ref}$ 5–15%.

### 32.2. Технические ограничения

- VK API rate 3 req/s.
- SQLite однопользовательский + ActiveSessionRegistry.
- Python-сидекар один; семафор 4.
- Один экземпляр приложения.
- Референс только для $I_{base}$.

### 32.3. Защитные механизмы

- Checkpointing на всех этапах, включая после $M_{reach}$ и после discovery-нормализации.
- Атомарные NLP-режимы.
- Интерактивный recovery UX с сохранением статуса PAUSED_PENDING_RECOVERY.
- Hard gates (5 условий).
- Блокировка рекомендаций при $Q < 0.5$, confidence < 0.25, NORM_UNSTABLE_SEVERE, CV_IQR > 0.5.
- Флаги качества.
- Single-instance + ActiveSessionRegistry.
- Flyway без downgrade.
- Soft-delete.
- AuditLog ~35 событий.
- Санитизация PII в логах с build-time проверкой.
- Защита GIANT от рутинных.
- Пересчёт нормализации после discovery.
- Backoff на сетевых операциях.
- Auto-restart Python (до 3) затем RecoveryDialog.

### 32.4. Допущения

1. VK-аккаунт с правами чтения публичных стен.
2. Открытые стены целевых сообществ.
3. Current ≥ 14 дней.
4. Python 3.10+ для FULL; иначе авто-FALLBACK.
5. Ресурсные файлы в дистрибутиве.

## 35. Итоговая end-to-end логика функционирования системы

Система решает прикладную задачу раннего предупреждения об информационных рисках через количественное, прозрачное и методологически обоснованное моделирование ЛОМ. Ключевые свойства:

1. **Разделение структурного и ситуационного** (двухмерная модель).
2. **Раздельные временные окна** (baseline vs current; тематический фильтр к обоим).
3. **Детерминированное разрешение** через матрицу ролей 4×2 с явными метками и модификаторами.
4. **Защита от артефактов** (GIANT с защитой от рутинных, сезонность с праздниками, γ-рассогласование с пересчётом, $k_{window}$-нормировка, Theil-Sen устойчивая к выбросам, Huber для тональности).
5. **Количественная неопределённость** (CI двухуровневый/блочный/бутстраповый bootstrap; confidence со штрафом за размер выборки; SQS с hard gates; флаги качества).
6. **Интерактивные режимы** (атомарные NLP с PythonRecoveryDialog при сбоях; сессии PAUSED_PENDING_RECOVERY при закрытии во время ожидания).
7. **Политика γ-рассогласования с тремя уровнями** (OK, MILD_RECOMPUTED с пересчётом $E_{norm}^{ref}$, AUDIENCE_ONLY_REFERENCE).
8. **Процедура расчёта τ^{ref}_{base}** (через самоориентированную Robust Sigmoid на референсной выборке с p75).
9. **Confidence = 0 для BASELINE_UNKNOWN** (по определению).
10. **Честное признание границ** ($I_{event}$ сессионно-относительно; stance не поддерживается; мультимодальный контент — только VAR; референс индикативен; допущение стационарности).
11. **Полная воспроизводимость** — все параметры, статистики, версии, переключения, флаги — в БД; ~35 событий AuditLog.
12. **Правовая и этическая встроенность** — шифрование токена, санитизация логов с build-time проверкой, soft-delete retention, WebView OAuth.
13. **Управление ресурсами** — SingleInstanceLock, ActiveSessionRegistry с состоянием RECOVERY_AWAITING, семафор Python.
14. **Обязательные версионированные ресурсы** — reference_base.json, holidays.json (с поддержкой нескольких версий для обратной совместимости), sentilex_base.json, test_corpus.json.
15. **Реалистичная feasibility** — NFR-03 90 мин с подтверждением benchmark-ом; параллелизация bootstrap; обеспеченный план дистрибуции.
16. **Имплементационная дорожная карта** — 8 спринтов с разбиением аналитического ядра на 4a/4b; явные gate-точки к релизу v1.0.

Система не претендует на автоматизацию управленческих решений. Её ценность — в структурировании информационного поля, количественной декомпозиции сигналов, раннем маркировании нестабильных выводов и прозрачной фиксации всех допущений.
